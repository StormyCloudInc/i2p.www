概述

该提案概述了一种适用于创建可盲化目的地的可重新随机化签名方案。此外，还可以用于盲化现有的 Ed25519 目的地，但效果稍有降低。

动机

Proposal 123 定义了一种加密的 LeaseSet2 格式，体现了最小授权原则：每个网络参与者仅获取角色所需的信息。特别是，发布到 floodfill 的加密 LeaseSet2 不会暴露它的目的地，并且只能由事先知道目的地的人查看。但是，floodfill 仍然需要能够在发布加密 LeaseSet2 时进行身份验证，客户端还需要确保身份验证是由目的地本身执行的。

Prop123 通过盲化目的地的签名密钥实现了这一点。被盲化的密钥可用于创建可由 floodfill 验证的签名，客户端可以确信只有目的地才能创建这些签名。因此，有必要指定一种可用于盲化的签名方案。

设计

核心签名方案

本提案中描述的签名方案 Red25519 是 RedDSA 的一个实例，它是一种支持密钥重新随机化的基于 Schnorr 的签名方案。它具有以下功能：

GENERATE_PRIVATE() 返回一个均匀随机的私钥。

DERIVE_PUBLIC(sk) 返回与给定私钥相对应的公钥。

GENERATE_RANDOM() 返回一个适合重新随机化密钥对的随机标量。

RANDOMIZE_PRIVATE(sk, alpha) 使用秘密标量 alpha 重新随机化私钥。

RANDOMIZE_PUBLIC(vk, alpha) 使用秘密标量 alpha 重新随机化公钥。

SIGN(sk, m) 返回由私钥 sk 对给定消息 m 的签名。

VERIFY(vk, m, sig) 验证与公钥 vk 和消息 m 对应的签名 sig。如果签名有效，返回 true；否则返回 false。

对于给定的密钥对 (sk, vk)，以下关系成立：

RANDOMIZE_PUBLIC(vk, alpha) == DERIVE_PUBLIC(RANDOMIZE_PRIVATE(sk, alpha))

将 Ed25519 密钥转换为 Red25519

Ed25519 密钥可暂时单向转换为 Red25519 密钥，以支持重新随机化现有的 Ed25519 目的地。其他签名类型不兼容。

我们定义以下转换函数：

CONVERT_ED25519_PRIVATE(privkey) 返回与给定 Ed25519 私钥相对应的 Red25519 私钥。

CONVERT_ED25519_PUBLIC(pubkey) 返回与给定 Ed25519 公钥相对应的 Red25519 公钥。

对于给定的 Ed25519 密钥对 (privkey, pubkey)，以下关系成立：

CONVERT_ED25519_PUBLIC(pubkey) == DERIVE_PUBLIC(CONVERT_ED25519_PRIVATE(privkey))

规格

定义

B RFC-8032 中的 Ed25519 基点。

L RFC-8032 中的 Ed25519 阶数 2^252 + 27742317777372353535851937790883648493。

[s] B 基点乘以 s 的固定基数标量乘法。

[s] A A 乘以 s 的可变基数标量乘法。

x || y 连接两个字节数组 x 和 y。

Red25519

方案 Red25519 专门化了 RedDSA，其中：

  • G := 曲线 Ed25519 形式的点群。具体而言，这意味着 Red25519 使用 L 阶的素数子群，且系数 h_G 为 8。
  • P_G := Ed25519 基点 B。
  • l_H := 512
  • H(x) := SHA-512(“I2P_Red25519H(x)” || x)

RedDSA 假设 H(x) 是使用对长度扩展攻击安全的加密哈希函数实例化的。SHA-512 本身不满足这一点。为了解决这个问题，我们要求消息以其长度的前缀自由编码为前缀：

len_u16(M) || M

其中 len_u16(M) 是 M 长度的 2 字节表示，采用小端序（与标量和点的小端编码一致）。

消息长度不得超过 65534 字节。65535 的长度保留用于可能的未来扩展。

编码和解码

Red25519 私钥是 L 模的标量，以小端序表示。我们定义 DECODE_SCALAR 和 ENCODE_SCALAR 函数用于在标量的字节数组和整数形式之间转换。

Red25519 公钥是曲线 Ed25519 形式上的点。它们被编码为 y 坐标的 255 位小端表示，后面是一个单个位指示 x 坐标的符号。这与 Ed25519 的编码相同。我们定义 DECODE_POINT 和 ENCODE_POINT 函数用于在字节数组和点的坐标形式之间转换。

RedDSA 函数

为了便于实现，我们在下面明确写出了已经为 Red25519 专门化的 RedDSA 函数（以及若干辅助函数）。实现者应参考 RedDSA 第 5.4.6 节了解 RedDSA 函数的一般规范。

HStar(prefix1, prefix2, m) :=
      h = SHA-512()
      h.input("I2P_Red25519H(x)")
      h.input(prefix1)
      h.input(prefix2)

      h.input(len(m) & 0xff)
      h.input((len(m) >> 8) & 0xff)
      h.input(m)

      s = h.digest()
      return s mod L

  GENERATE_PRIVATE :=
      s = 64 random bytes
      return s mod L

  DERIVE_PUBLIC(sk) := [sk] B

  GENERATE_RANDOM :=
      s = 64 random bytes
      return s mod L

  RANDOMIZE_PRIVATE(sk, alpha) := (sk + alpha) mod L

  RANDOMIZE_PUBLIC(vk, alpha) := vk + [alpha] B

  SIGN(sk, m) :=
      T = 80 random bytes
      vkBytes = ENCODE_POINT(DERIVE_PUBLIC(sk))
      r = HStar(T, vkBytes, m)

      R = [r] B
      Rbytes = ENCODE_POINT(R)

      c = HStar(Rbytes, vkBytes, m)
      S = (r + (c * sk)) mod L

      return Rbytes || ENCODE_SCALAR(S)

  VERIFY(vk, m, sig) :=
      Rbytes = sig[0..32]
      Sbytes = sig[32..64]

      R = DECODE_POINT(Rbytes)
      if R is invalid:
          return false

      S = DECODE_SCALAR(Sbytes)
      if S >= L:
          return false

      vkBytes = ENCODE_POINT(vk)
      c = HStar(Rbytes, vkBytes, m)
      return ((-[S] B) + R + ([c] vk)).multiplyByCofactor().isIdentity()

转换函数

CONVERT_ED25519_PRIVATE(privkey) :=
      s = SHA-512(privkey)[0..32]
      s[0] = s[0] & 248
      s[31] = (s[31] & 63) | 64
      return s

  CONVERT_ED25519_PUBLIC(pubkey) := pubkey

注意，CONVERT_ED25519_PRIVATE 的实现等同于从 RFC-8032 第 5.1.5 节步骤 1-3 中的 Ed25519 私钥推导 Ed25519 公钥时的秘密标量 s 的计算。

安全影响

重新随机化 Red25519 目的地然后使用它创建签名不会泄露任何关于目的地的信息，因为通过 RANDOMIZE_PRIVATE 生成的 Red25519 私钥的分布与通过 GENERATE_PRIVATE 生成的私钥的分布相同，并且 DERIVE_PUBLIC 是确定性的。

通过 CONVERT_ED25519_PRIVATE 将 Ed25519 私钥转换为 Red25519 不会产生相同的分布。然而，我们认为安全性降低是可以接受的，原因如下：

  • Ed25519 标量空间大约是 Red25519 标量空间的一半（大约有 2^251 个可能的 Ed25519 标量，而 L ~= 2^252 个可能的 Red25519 标量）。因此，安全性损失最多不过是大约 2 倍，即大约 1 位（因为我们可能碰巧选择了一个既是 Red25519 标量，又是有效的 Ed25519 标量）。

  • 现有的 Ed25519 目的地已经在网络上历史性地暴露，应该假设恶意的 floodfill 已经枚举过它们。

对该安全降低感到担忧的用户应使用 Red25519 作为其目的地的签名类型，而非 Ed25519。

注意，上述论点不适用于重新随机化标量 alpha；每次选择有偏见的 alpha 时，密钥的信息都会泄露，因为加法重新随机化行为类似于一次性密钥。

兼容性

支持 Red25519 的 I2P 版本将能够验证使用它签名的网络数据结构。不支持 Red25519 的 I2P 版本将把它视为未知签名，并可以丢弃该数据结构。在 Red25519 的充分部署得到实现之前，用户应该预期 Red25519 签名的数据结构的可靠性较差。
