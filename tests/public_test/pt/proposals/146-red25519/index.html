Nota

Implantação e testes na rede em andamento. Sujeito a pequenas revisões. Veja specification para a especificação oficial.

Visão geral

Esta proposta descreve um esquema de assinatura re-randomizável adequado para criar Destinations que podem ser cegadas. Adicionalmente, pode ser usado para cegar Destinations Ed25519 existentes, com uma ligeira redução na eficácia.

Motivação

Proposal 123 define um formato criptografado LeaseSet2 que incorpora o Princípio de Menor Autoridade: cada participante da rede recebe apenas as informações necessárias para seu papel. Em particular, um LeaseSet2 criptografado publicado em um floodfill não revela o Destination para o qual é destinado, e os Leases só podem ser vistos por alguém com conhecimento prévio do Destination. No entanto, os floodfills ainda precisam autenticar os LeaseSet2s criptografados quando eles são publicados, e os clientes precisam garantir adicionalmente que a autenticação foi imposta pelo próprio Destination.

Prop123 alcança isso cegando as chaves de assinatura dos Destinations. As chaves cegadas podem ser usadas para criar assinaturas que são verificáveis pelos floodfills, e os clientes podem ter certeza de que apenas o Destination poderia ter criado as assinaturas. Portanto, é necessário especificar um esquema de assinatura que possa ser usado para cegamento.

Design

Esquema de assinatura principal

O esquema de assinatura descrito nesta proposta, Red25519, é uma instância do RedDSA , um esquema de assinatura baseado em Schnorr que suporta re-randomização de chaves. Ele tem as seguintes funções:

GENERATE_PRIVATE() Retorna uma chave privada uniformemente aleatória.

DERIVE_PUBLIC(sk) Retorna a chave pública correspondente à chave privada dada.

GENERATE_RANDOM() Retorna um escalar aleatório adequado para re-randomizar um par de chaves.

RANDOMIZE_PRIVATE(sk, alpha) Re-randomiza uma chave privada, usando um escalar secreto alpha.

RANDOMIZE_PUBLIC(vk, alpha) Re-randomiza uma chave pública, usando um escalar secreto alpha.

SIGN(sk, m) Retorna uma assinatura da chave privada sk sobre a mensagem dada m.

VERIFY(vk, m, sig) Verifica a assinatura sig contra a chave pública vk e a mensagem m. Retorna true se a assinatura for válida, false caso contrário.

Para um par de chaves dado (sk, vk) a seguinte relação mantém-se:

RANDOMIZE_PUBLIC(vk, alpha) == DERIVE_PUBLIC(RANDOMIZE_PRIVATE(sk, alpha))

Convertendo chaves Ed25519 para Red25519

Chaves Ed25519 PODEM ser transitoriamente convertidas de maneira unilateral em chaves Red25519, a fim de suportar a re-randomização de Destinations Ed25519 existentes. Outros tipos de assinatura não são compatíveis.

Definimos as seguintes funções de conversão:

CONVERT_ED25519_PRIVATE(privkey) Retorna a chave privada Red25519 correspondente à chave privada Ed25519 dada.

CONVERT_ED25519_PUBLIC(pubkey) Retorna a chave pública Red25519 correspondente à chave pública Ed25519 dada.

Para um par de chaves Ed25519 dado (privkey, pubkey) a seguinte relação mantém-se:

CONVERT_ED25519_PUBLIC(pubkey) == DERIVE_PUBLIC(CONVERT_ED25519_PRIVATE(privkey))

Especificação

Definições

B O ponto base Ed25519 como em RFC-8032 .

L A ordem Ed25519 2^252 + 27742317777372353535851937790883648493, como em RFC-8032 .

[s] B Multiplicação de escalar de base fixa do ponto base por s.

[s] A Multiplicação de escalar de base variável de A por s.

x || y Concatenação de duas arrays de bytes x e y.

Red25519

O esquema Red25519 especializa RedDSA com:

  • G := o grupo de pontos na forma Edwards do Curve25519. Em particular, isso significa que o Red25519 usa o subgrupo de ordem primo de ordem L, e o cofator h_G é 8.
  • P_G := o ponto base Ed25519 B.
  • l_H := 512
  • H(x) := SHA-512(“I2P_Red25519H(x)” || x)

RedDSA assume que H(x) é instanciado com uma função de hash criptográfica que é segura contra ataques de extensão de comprimento. O SHA-512 não satisfaz isso por conta própria. Para remediar isso, exigimos que as mensagens sejam prefixadas com uma codificação livre de prefixo do seu comprimento:

len_u16(M) || M

onde len_u16(M) é a representação de 2 bytes do comprimento de M, em little-endian (para ser consistente com a codificação little-endian de escalares e pontos).

As mensagens não devem ter mais de 65534 bytes. Um comprimento de 65535 é reservado para possíveis extensões futuras.

Codificação e decodificação

As chaves privadas Red25519 são escalares mod L, codificadas em representação little-endian. Definimos as funções DECODE_SCALAR e ENCODE_SCALAR para a troca entre a forma de array de bytes e a forma de inteiro de um escalar.

As chaves públicas Red25519 são pontos na forma Edwards do Curve25519. Elas são codificadas como a representação little-endian de 255 bits da coordenada y, seguida por um único bit indicando o sinal da coordenada x. Esta é a mesma codificação que para Ed25519. Definimos as funções DECODE_POINT e ENCODE_POINT para a troca entre a forma de array de bytes e forma de coordenadas de um ponto.

Funções de RedDSA

Para facilitar a implementação, escrevemos explicitamente abaixo as funções RedDSA (assim como várias funções auxiliares) já especializadas para Red25519. Os implementadores devem consultar a seção 5.4.6 de RedDSA para a especificação geral das funções RedDSA.

HStar(prefix1, prefix2, m) :=
      h = SHA-512()
      h.input("I2P_Red25519H(x)")
      h.input(prefix1)
      h.input(prefix2)

      h.input(len(m) & 0xff)
      h.input((len(m) >> 8) & 0xff)
      h.input(m)

      s = h.digest()
      return s mod L

  GENERATE_PRIVATE :=
      s = 64 random bytes
      return s mod L

  DERIVE_PUBLIC(sk) := [sk] B

  GENERATE_RANDOM :=
      s = 64 random bytes
      return s mod L

  RANDOMIZE_PRIVATE(sk, alpha) := (sk + alpha) mod L

  RANDOMIZE_PUBLIC(vk, alpha) := vk + [alpha] B

  SIGN(sk, m) :=
      T = 80 random bytes
      vkBytes = ENCODE_POINT(DERIVE_PUBLIC(sk))
      r = HStar(T, vkBytes, m)

      R = [r] B
      Rbytes = ENCODE_POINT(R)

      c = HStar(Rbytes, vkBytes, m)
      S = (r + (c * sk)) mod L

      return Rbytes || ENCODE_SCALAR(S)

  VERIFY(vk, m, sig) :=
      Rbytes = sig[0..32]
      Sbytes = sig[32..64]

      R = DECODE_POINT(Rbytes)
      if R is invalid:
          return false

      S = DECODE_SCALAR(Sbytes)
      if S >= L:
          return false

      vkBytes = ENCODE_POINT(vk)
      c = HStar(Rbytes, vkBytes, m)
      return ((-[S] B) + R + ([c] vk)).multiplyByCofactor().isIdentity()

Funções de conversão

CONVERT_ED25519_PRIVATE(privkey) :=
      s = SHA-512(privkey)[0..32]
      s[0] = s[0] & 248
      s[31] = (s[31] & 63) | 64
      return s

  CONVERT_ED25519_PUBLIC(pubkey) := pubkey

Note que a implementação de CONVERT_ED25519_PRIVATE é equivalente ao cálculo do escalar secreto s ao derivar uma chave pública Ed25519 de uma chave privada Ed25519, conforme especificado nos passos 1-3 da seção 5.1.5 de RFC-8032 .

Implicações de segurança

Re-randomizar um Destination Red25519 e depois criar assinaturas com ele não revela nenhuma informação sobre o Destination, porque a distribuição de chaves privadas Red25519 geradas via RANDOMIZE_PRIVATE é idêntica à distribuição de chaves privadas geradas via GENERATE_PRIVATE, e DERIVE_PUBLIC é determinística.

Converter chaves privadas Ed25519 para Red25519 via CONVERT_ED25519_PRIVATE não resulta na mesma distribuição. No entanto, consideramos a redução na segurança aceitável pelos seguintes motivos:

  • O espaço dos escalares Ed25519 é aproximadamente metade do tamanho do espaço dos escalares Red25519 (há 2^251 possíveis escalares Ed25519, e L ~= 2^252 possíveis escalares Red25519). Portanto, a perda de segurança é, no máximo, um fator de aproximadamente 2, ou aproximadamente 1 bit (porque poderíamos ter escolhido coincidentemente um escalar Red25519 que também é um escalar Ed25519 válido).

  • Destinations Ed25519 existentes já foram historicamente expostas na rede, e deve-se assumir que floodfills maliciosos já as enumeraram.

Usuários que estão preocupados com essa redução de segurança devem usar Red25519 como o tipo de assinatura para seus Destinations em vez de Ed25519.

Note que o argumento acima não se aplica ao escalar de re-randomização alpha; informações sobre a chave são reveladas cada vez que um alpha tendencioso é escolhido, pois a re-randomização aditiva se comporta como um cifra de bloco única.

Compatibilidade

Versões do I2P que suportam Red25519 serão capazes de verificar estruturas de dados da rede que são assinadas com ele. Versões do I2P que não suportam Red25519 tratarão como uma assinatura desconhecida e PODERÃO descartar as estruturas de dados. Os usuários devem esperar que a confiabilidade das estruturas de dados assinadas por Red25519 seja ruim até que uma implantação suficiente do Red25519 seja alcançada.
